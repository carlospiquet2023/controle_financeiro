# Cloudflare R2 no Finora

Guia operacional fiel à integração atual. O bucket deve ser privado.

## Fluxos que usam R2

| Fluxo       | Upload                         | Persistência                       | Leitura/exclusão                            |
| ----------- | ------------------------------ | ---------------------------------- | ------------------------------------------- |
| Excel/CSV   | servidor (`PutObject`)         | `ImportBatch.sourceKey/sourceHash` | original preservado; não há download na UI  |
| comprovante | navegador por URL PUT assinada | `Attachment` após PUT              | download autenticado e exclusão disponíveis |
| XML fiscal  | servidor (`PutObject`)         | `TaxDocument.sourceKey/sourceHash` | preservado; removido no reset total         |
| OCR         | lê o comprovante privado       | campos ficam em `Attachment`       | não cria novo objeto                        |

## Variáveis

```dotenv
R2_ACCOUNT_ID=""
R2_ACCESS_KEY_ID=""
R2_SECRET_ACCESS_KEY=""
R2_BUCKET="finora-receipts"
R2_PUBLIC_URL=""
```

`R2_PUBLIC_URL` não é lida pelo código. O servidor usa URLs GET assinadas; não torne o bucket público.

Endpoint S3:

```text
https://{R2_ACCOUNT_ID}.r2.cloudflarestorage.com
```

Região: `auto`.

## Prefixos

```text
households/{householdId}/imports/{YYYY-MM-DD}/{uuid}.{xlsx|xls|csv}
households/{householdId}/receipts/{uuid}.{jpg|jpeg|png|webp|pdf}
households/{householdId}/tax/{YYYY-MM-DD}/{uuid}.xml
```

O prefixo ajuda no isolamento e limpeza, mas a autorização real continua no servidor e na política do bucket.

## Permissões recomendadas

Crie credencial exclusiva por ambiente, limitada ao bucket, com operações necessárias de objeto. O Finora usa `PutObject`, `GetObject` e `DeleteObject`. Não reutilize token administrativo da conta Cloudflare.

## Importação de planilha

`POST /api/import-file`:

- permite OWNER, ADMIN e EDITOR;
- aceita `.xlsx`, `.xls` e `.csv` até 10 MB;
- calcula SHA-256;
- grava metadata `sha256` e nome original codificado;
- retorna key/hash para a Server Action.

O lote só é criado após revalidação do payload. O servidor ainda não baixa o original para comprovar que as linhas normalizadas correspondem ao arquivo. Rollback e reset financeiro não apagam originais de importação; configure lifecycle/retention.

## Comprovantes

`POST /api/upload-url` recebe nome, MIME e tamanho e devolve URL PUT válida por 300 segundos. Limite: 10 MB. Tipos:

- `image/jpeg`;
- `image/png`;
- `image/webp`;
- `application/pdf`.

O navegador usa o mesmo `Content-Type` no PUT e chama `registerAttachment`. O servidor valida que a transação pertence à família e que a key começa pelo prefixo esperado.

Download: `GET /api/attachments/[id]/download` gera URL assinada por 120 segundos com nome seguro. Exclusão remove objeto e registro. OCR busca o mesmo objeto diretamente pelo SDK.

Reset mensal/total tenta apagar comprovantes. A limpeza usa melhor esforço: falha R2 não reverte a transação PostgreSQL, então objetos órfãos ainda precisam de rotina operacional.

## XML fiscal

`POST /api/tax/documents/import` aceita XML até 10 MB, calcula hash, bloqueia duplicidade dentro da família e preserva o original antes de criar `TaxDocument` e itens. Reset total tenta excluir essas keys; não existe download de XML na interface.

## CORS

Uploads server-side de planilha/XML não precisam de CORS. O comprovante usa PUT direto do navegador e exige regra restrita ao domínio:

```json
[
  {
    "AllowedOrigins": ["http://localhost:3000", "https://seu-dominio.example"],
    "AllowedMethods": ["PUT"],
    "AllowedHeaders": ["Content-Type"],
    "ExposeHeaders": ["ETag"],
    "MaxAgeSeconds": 300
  }
]
```

Não use `*` em produção.

## Retenção e incidentes

- defina lifecycle para originais antigos/órfãos;
- separe buckets ou credenciais por ambiente;
- restrinja acesso administrativo e registre intervenções;
- não coloque chave, URL assinada ou documento em logs/issues;
- rotacione imediatamente qualquer credencial exposta;
- mantenha backup e política de exclusão compatíveis com dados financeiros/fiscais.

## Diagnóstico

| Sintoma                         | Verifique                                        |
| ------------------------------- | ------------------------------------------------ |
| “armazenamento não configurado” | Account ID e credenciais                         |
| “bucket não configurado”        | `R2_BUCKET`                                      |
| PUT 403                         | CORS, `Content-Type`, expiração e relógio        |
| download falha                  | key ainda existe e credencial possui `GetObject` |
| exclusão deixa órfão            | permissão `DeleteObject` e logs Cloudflare       |
| importação para antes do banco  | R2 é obrigatório no fluxo web                    |

`/api/health/ready` testa apenas PostgreSQL, não R2.
